# 刷到即拥有：短视频购买意图压缩器

## Swipe-to-Own · 从“我想要”到“可买候选”

> **一句话定义：** Swipe-to-Own 是一个把短视频里的模糊购买冲动，压缩成可点击、可比较、可购买候选的交互系统。
>
> **比赛版边界：** 只做服饰穿搭 Buy MVP：精选短视频 + 本地商品 catalog + 点选服饰 + Top-3 相似可购候选 + 一次 refine + 外跳详情。
>
> **不承诺：** 不承诺 exact SKU，不做全品类交易，不把 Do / Go / Know 作为主线，不让 AI 替用户做购买决定。

---

# 1. 问题：短视频里的购买意图高冲动、低表达

用户在短视频里看到喜欢的衣服、鞋包、穿搭时，经常不是带着明确搜索词来的。他们只是在某一帧产生了一个模糊冲动：

**“这个感觉我也想要。”**

但今天的路径很长：

```text
暂停视频 -> 截图 -> 切到电商/搜索/小红书 -> 猜关键词 -> 上传图片或手搜 -> 筛结果 -> 再判断像不像、贵不贵、能不能买
```

这个过程的问题不是“识别不了这是什么”，而是：

| 断点 | 用户损失 |
|---|---|
| 需要截图和切 App | 内容语境断了，兴趣变冷 |
| 不知道怎么描述 | 关键词、版型、风格说不清 |
| 搜图结果太散 | 用户还要自己比价和筛选 |
| 同款预期太高 | 找不到 exact SKU 就容易觉得失败 |
| 决策证据不足 | 不知道为什么这些候选值得点 |

Swipe-to-Own 要证明的命题不是“AI 能认出衣服”，而是：

**短视频里的瞬时种草意图，可以被压缩成用户愿意点击的购买候选。**

---

# 2. 定义：什么叫购买意图压缩器

购买意图压缩器做三件事：

| 层级 | 输入 | 压缩结果 |
|---|---|---|
| 对象压缩 | 当前视频帧 + 用户点选区域 | “用户想找的是哪件服饰” |
| 约束压缩 | 预算、颜色、风格、场景等可选表达 | “用户在意什么条件” |
| 行动压缩 | 商品供给、相似度、价格、来源 | “3 个可点击比较的候选” |

最终输出不是百科解释，也不是聊天回答，而是一张行动卡：

```text
我理解你想找：米色长款通勤外套
这里有 3 个相似可买候选：
- 为什么像
- 哪里不完全满足
- 价格和来源
- 查看商品
```

AI 的角色是辅助比较，不替用户决定。它把“说不清的喜欢”转成“可确认的选择”。

# 3. MVP：只验证服饰 Buy 单闭环

## 3.1 P0 用户路径

```text
刷短视频
  -> 看到喜欢的服饰
  -> 点右侧「我也要」
  -> 当前帧冻结
  -> point / box 选中衣服、鞋或包
  -> 可选补充：更便宜 / 同风格 / 更日常 / 换颜色
  -> 返回 Top-3 相似可购买候选
  -> 查看商品 / 收藏 / 换一批
```

## 3.2 首版只服务一种用户

**已有轻度购买意图的服饰种草用户。**

他们不是来泛问知识，也不是来完整规划穿搭，而是在内容消费中被某个单品打动，想快速知道：

1. 有没有类似的。
2. 贵不贵。
3. 像在哪里。
4. 能不能点进去看。

## 3.3 首版限定场景

| 做 | 不做 |
|---|---|
| 女装上衣、外套、连衣裙等高频服饰 | 全品类商品 |
| 主体清晰、服饰可见、遮挡少的视频 | 多人混乱、强遮挡、低清、快速转场作为主路径 |
| 相似款、同风格、预算友好候选 | 稳定 exact 同款识别 |
| 单品购买候选 | 整套穿搭自动购买 |
| 一次 refine | 多轮聊天式导购 |

Do / Go / Know 只保留为未来行动卡扩展方向，不进入本次验收。

# 4. 体验：3 步完成从种草到候选

## 4.1 入口

主入口：右侧栏「我也要」。

辅助入口：长按画面快捷冻结。

普通点击仍保留短视频习惯语义，例如暂停/播放，避免入口混乱。

入口不是强广告位。它只在用户停留、暂停、回看等高意图信号出现后增强提示，避免打断内容消费。

## 4.2 点选

用户点「我也要」后，视频冻结当前帧。

P0 只支持两种选取：

| 方式 | 用途 |
|---|---|
| point | 默认，点一下目标服饰 |
| box | 目标较小或重叠时兜底 |

点选后系统先确认对象：

```text
已选择：米色长款外套
正在找相似可买款
```

如果低置信度，先澄清：

```text
你想找的是外套、包，还是鞋？
```

## 4.3 候选卡

首屏只给 3 个候选，降低决策负担；同时提供“查看更多相似款”。

每张卡只展示决策必需信息：

| 字段 | 作用 |
|---|---|
| 商品图 | 让用户判断像不像 |
| 价格 | 判断是否符合预算 |
| 来源 | 建立可信度 |
| 相似点 | 颜色、版型、风格、价格带等 |
| 未满足点 | 材质未知、颜色略深、价格略高等 |
| 置信度 | 疑似同款 / 风格近似 / 条件不明 |
| CTA | 查看商品 |

文案统一使用“相似款”“相似风格”“可替代选择”，不使用“原款”“复刻”“必买”等暗示或催促性表达。

# 5. 机制：从画面到 Top-3 的压缩链路

## 5.1 最小链路

```text
前端抽帧 + 用户选区
  -> 生成视觉/约束查询
  -> 属性归一化
  -> catalog 检索
  -> hard filter + soft rerank
  -> Top-3 商品卡
```

## 5.2 查询协议

一次查询由不可变 `request_id` 表示。refine 会生成新的 `request_id`，并通过 `parent_request_id` 关联原请求。

核心 query schema：

| 字段 | 说明 |
|---|---|
| video_id | 当前视频 |
| timestamp_ms | 当前帧时间 |
| region | point 或 box |
| category | 主类目 + 备选类目 + 置信度 |
| explicit_constraints | 用户显式约束，如预算、颜色、风格 |
| visual_attributes | 视觉抽取属性，如颜色、长度、廓形 |
| normalized_tags | 映射到 catalog 的可检索标签 |
| uncertainty | 不确定属性及来源 |

## 5.3 过滤与排序

| 层级 | 规则 |
|---|---|
| hard filter | 类目底线、明确预算上限、不可售状态 |
| soft rerank | 颜色、版型、风格、价格接近度、相似标签 |
| 不确定属性 | 不进硬过滤，只低权重参与排序或解释 |

属性优先级：

```text
用户显式约束 > 高置信视觉属性 > 视频上下文 > 系统推断
```

类目不是单点硬死。低置信度时允许“主类目 + 备选类目”多路召回，再由 rerank 排序。

## 5.4 供给策略

demo catalog 不是随便凑商品，而是实验装置。

必须覆盖：

| 维度 | 例子 |
|---|---|
| 类目 | 上衣、外套、连衣裙 |
| 颜色族 | 米色、黑色、白色、蓝色 |
| 版型 | 宽松、短款、长款、修身 |
| 风格 | 通勤、休闲、简约、甜美 |
| 价格带 | 100-200、200-300、300-500 |

P0 用精选 catalog 保证链路稳定；生产版再接平台商品池、联盟导购或商家供给。

# 6. 验证：压缩是否真的成功

## 6.1 北极星问题

**用户原本说不清自己想买什么，系统是否把这个冲动压缩成了可执行购买决策？**

## 6.2 北极星指标

**L3 命中后的有效商品行动率。**

定义：当候选被判定为“购买可接受”后，用户发生以下任一行为：

- 查看商品详情；
- 收藏候选；
- 外跳商品页；
- 明确反馈“像 / 可接受”。

## 6.3 Benchmark 分层

| 层级 | 含义 | 用途 |
|---|---|---|
| L1 类别命中 | 返回的类目对不对 | 诊断识别底线 |
| L2 风格命中 | 颜色、版型、风格是否接近 | 诊断相似理解 |
| L3 购买可接受 | 用户是否愿意点进详情或纳入候选 | 主评测指标 |

L3 分三档：

| 档位 | 定义 |
|---|---|
| 可直接购买 | 用户认为很接近，愿意进入详情认真看 |
| 可接受替代 | 不是同款，但符合想要的感觉或预算 |
| 不可接受 | 品类、风格、价格或关键属性偏离 |

## 6.4 对照实验

基线：截图搜图 / 关键词搜索。

比较两类价值：

| 类型 | 指标 |
|---|---|
| 入口效率 | 点击数、耗时、是否需要截图、是否需要输入关键词 |
| 结果质量 | Top1/Top3/Top5 L3 命中率、收藏率、外跳率、不满意率 |

成功不能只看点击。必须满足：

```text
主指标提升 + 不满意率不恶化 + 内容侧护栏不恶化
```

内容侧护栏包括完播率、跳出率、不感兴趣反馈，避免交易入口劫持内容体验。

# 7. 风险与边界

## 7.1 产品风险

| 风险 | 降级策略 |
|---|---|
| 用户期待同款 | 入口和卡片明确“相似款/替代款” |
| 点错目标 | 高亮目标，可重选 |
| 多目标歧义 | 先问“你想找外套、包还是鞋？” |
| 候选不像 | 显示未满足点，支持换一批或一次 refine |
| 结果过少 | 首屏 Top-3 + 查看更多相似款 |

## 7.2 技术风险

| 风险 | 降级策略 |
|---|---|
| canvas 跨域污染 | demo 视频自托管并配置 CORS |
| 外部 API 慢 | 先返回本地 catalog |
| SSE 不稳 | 可降级为一次性响应 |
| 类目识别低置信 | 多类目召回 + 用户确认 |
| 商品供给不足 | 明确 catalog 覆盖范围，不装作全网可买 |

## 7.3 隐私与版权

| 边界 | 约定 |
|---|---|
| 视频素材 | demo 只用自制或授权素材 |
| 原视频 | 默认不存原视频 |
| 临时帧 | 优先端侧裁剪后上传；短期缓存，TTL 删除 |
| 用户约束 | 只保存结构化约束，不保留不必要原始语音 |
| 人脸与身份 | 不做人脸识别、不做身份判断、不做人群画像 |
| 商业排序 | 广告/合作/佣金影响必须显著标注 |

# 8. 商业化：先证明高意图，再谈规模

短期商业模式：CPS / 联盟导购。

但点击和外跳只是前置指标。真正商业闭环是：

```text
有效外跳 -> 下单 -> 佣金确认 -> 退款后净佣金
```

自然候选排序不因佣金率改变。商业合作只能进入显著标识的广告位/合作位，不能混入自然候选伪装成中立推荐。

P0 不承诺 GMV，只证明：

1. 用户愿意在视频内触发购买意图；
2. 系统能把模糊意图转成可接受候选；
3. 用户愿意进入商品详情或收藏候选。

# 9. 交付策略

## 9.1 必做

| 模块 | P0 标准 |
|---|---|
| Feed | 竖滑视频，当前视频稳定播放 |
| 入口 | 右侧「我也要」主入口 |
| 冻结点选 | 点入口冻结，point/box 选目标 |
| 抽帧 | 当前帧 + region 生成查询 |
| catalog | 本地精选商品库 |
| 商品卡 | Top-3，相似点、未满足点、价格、来源、CTA |
| refine | 最多一次：更便宜、同风格、更日常、换颜色 |
| 评测 | 20-30 条样本，L1/L2/L3 标注 |

## 9.2 可砍

| 能力 | 处理 |
|---|---|
| 语音 | P0 可砍，文本/chips 代替 |
| SAM2 | P0 可砍，point/box 代替 |
| Do / Go / Know | 不进主 demo |
| 账号系统 | 不做 |
| 复杂预解析 | 可用手工 JSON 或本地标签代替 |
| 真实全网搜索 | 不作为 P0 依赖 |

## 9.3 最小演示版

如果只剩最后半天，保留：

```text
5 条精选视频
+ 1 个右侧入口
+ point 选服饰
+ 本地 catalog Top-3
+ 商品卡可点击
+ 1 个对照实验结果
```

这已经能证明核心命题，不需要堆更多 AI 能力。

# 10. 答辩 Q&A

## Q1：这不就是识图购物吗？

不是。识图购物从一张图出发，返回相似商品。Swipe-to-Own 从短视频里的种草瞬间出发，让用户在视频内点选局部对象，补充预算/风格约束，并返回可比较候选。我们压缩的是购买意图，不只是识别图片。

## Q2：不承诺同款，为什么用户会信？

因为我们不把相似款包装成同款。卡片会明确展示相似点、未满足点、来源和置信度。评估重点不是同款命中，而是用户是否认为候选可接受、愿意点进详情或收藏。

## Q3：小 catalog 是不是作秀？

P0 目标不是证明全网覆盖，而是证明链路成立：用户愿意触发，系统能返回可接受候选，候选能产生商品行动。catalog 扩大是供给规模问题，不改变核心交互和指标框架。

## Q4：怎么证明真的带来购买，而不是好看的按钮？

看三层证据：入口效率是否减少步骤和时间；Top-3 L3 命中率是否足够高；L3 命中后是否产生详情查看、收藏、外跳等有效商品行动。

## Q5：视频里有很多东西，系统怎么知道用户想买哪件？

不让系统猜。用户点「我也要」后再点选具体服饰或区域。低置信度时先澄清目标，再返回候选。复杂识别问题被转化为低成本的人机协同选择。

# 11. 最终记忆点

Swipe-to-Own 不是万能 AI 按钮，也不是传统识图购物。

它是：

**短视频里的购买意图压缩器。**

它把一句模糊的“我也想要”，压缩成三张可比较、可确认、可点击的购买候选卡。
